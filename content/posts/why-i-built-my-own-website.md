我一直知道自己需要一個可以長期累積內容的地方，但「開始寫」和「建立自己的網站」其實是兩個不同的決定。

如果只是想立刻發表文章，我可以選方格子、Medium 或 Blogger；如果想要完整的內容管理後台，也可以租主機架 WordPress。這些方案都比從零建立網站更快，也各有成熟的讀者、工具或生態系。

但我最後選擇了另一條路：用 **Astro 建立網站、以 GitHub 保存內容，再透過 GitHub Pages 發布**。這個網站不只是一個部落格，也同時是我的作品集，以及我練習如何與 AI 協作完成真實專案的第一件作品。

> 簡單說，我不是只想找一個「可以貼文章的地方」，而是想建立一個能由自己決定結構、視覺與未來方向的數位基地。

## 我真正要解決的問題

一開始，我問的不是「哪個平台最好」，而是「我希望這個地方未來可以做什麼」。整理之後，我需要的是：

1. 可以持續發表文章，而且內容容易搬移與保存。
2. 可以展示作品，不必把部落格和作品集拆成兩個網站。
3. 視覺能延續我自己的品牌，而不是被平台版型限制。
4. 網站要快、乾淨，並具備基本 SEO、RSS 與社群分享資訊。
5. 日後可以綁定自己的網域，不必重做網站。
6. 我不一定要自己寫程式，但要能透過 AI 協作維護。

當需求變成這六點，選擇就不再只是「哪裡寫文章最方便」，而是「我要租用別人的空間，還是建立自己的空間」。

## 方格子、Medium、WordPress、Blogger 與 Astro 怎麼選？

以下比較不是要分出絕對高下，而是從我這次「個人網站＋部落格＋作品集」的需求出發。

| 方案 | 最適合的情境 | 優點 | 需要接受的限制 |
| --- | --- | --- | --- |
| 方格子 vocus | 想快速寫作、經營華文讀者與訂閱變現 | 有編輯器、會員、互動、數據與變現工具，不必處理架站 | 品牌與頁面結構仍在平台框架內，作品集客製彈性較少 |
| Medium | 想用簡潔編輯器接觸國際讀者 | 閱讀體驗成熟，也有 publication 與既有內容網路 | 客製外觀與追蹤工具受限；自訂網域需要有效會員資格 |
| WordPress 自架 | 想要傳統後台、外掛與高度擴充 | 後台成熟，主題與外掛眾多，可做會員、商店或複雜內容網站 | 需要主機、更新、備份與安全維護；外掛越多，管理成本通常越高 |
| Blogger | 想免費、快速建立傳統部落格 | Google 帳號即可管理，支援自訂網域、HTTPS、Analytics 與內容匯入匯出 | 設計與現代開發彈性有限，作品集呈現較難完全客製 |
| Astro＋GitHub Pages | 想擁有程式碼、內容與品牌結構 | 速度快、攻擊面小、可高度客製；文章與作品可用內容集合管理 | 沒有傳統後台，需要 GitHub、編輯器或 AI 協助更新 |

方格子官方將自己定位為包含 CMS、會員互動、數據與多元變現工具的創作平台；這對想把心力集中在寫作與社群經營的人很有吸引力。[方格子創作者介紹](https://vocus.cc/become_creator/)

Medium 的 publication 具有首頁版面、電子報與協作功能，但即使連接自訂網域，網站仍在 Medium 平台內；官方也說明自訂網域需要有效的 Medium 會員資格，而且不能自行加入 Google Analytics 或其他程式碼。[Medium 自訂網域說明](https://help.medium.com/hc/en-us/articles/115003053487-Setting-up-a-custom-domain-for-your-profile-or-publication)

WordPress 的強項是成熟後台、主題與龐大的外掛系統，官方也強調它能透過外掛、自訂內容類型與 API 擴充。[WordPress 功能介紹](https://wordpress.org/about/features/) 但對我現在的需求來說，資料庫、主機和外掛維護不是最想投入的地方。

Blogger 則依然是低門檻的選擇：官方後台可以管理網址、搜尋引擎可見性、HTTPS、Analytics 與文章 feed，也能連接自訂網域。[Blogger 設定說明](https://support.google.com/blogger/answer/9691230?hl=zh-Hant)

最後，Astro 本來就是以內容型網站為核心，預設減少傳送到瀏覽器的 JavaScript，並提供內容集合來整理與驗證 Markdown 資料。[為什麼選 Astro？](https://docs.astro.build/en/concepts/why-astro/) 對一個以文章和作品為主的網站，這個取向剛好吻合。

## 為什麼我最後沒有直接選平台？

平台最大的優點，是一註冊就能開始；最大的代價，則是我的內容會活在平台設計好的關係裡。

這不代表使用平台就不擁有內容，而是網址、讀者路徑、頁面結構、推薦機制與視覺呈現，多少都要配合平台規則。對純寫作而言，這個交換往往非常合理；但我還希望同一個地方能承載作品、個人介紹與之後尚未決定的功能。

所以我的選擇不是拒絕方格子或 Medium。未來我仍然可以把文章摘要或精選內容同步到其他平台，把它們當成接觸讀者的入口；完整內容與作品檔案則以自己的網站作為主要版本。

這樣的策略比較像：**平台負責分發，個人網站負責累積。**

## 為什麼不是 WordPress？

WordPress 很適合需要多人登入、頻繁發文、會員系統、留言、電子商務或大量外掛功能的網站。它的後台讓不碰程式的人也能管理內容，這是明確優勢。

但我目前只有一位作者，主要內容是文章與作品。為了這些需求維護資料庫、主機、佈景主題和外掛，對我來說比較像是先買了一套大型工具，再只使用其中一小部分。

Astro 產生的是預先建好的靜態頁面。讀者打開網站時，不需要等待伺服器查詢資料庫再組合內容；網站也沒有公開登入後台，因此少了一部分需要長期防護的入口。代價是我不能像 WordPress 一樣登入 `/wp-admin` 後直接發文。

我接受這個取捨，因為現在可以讓 AI 成為我的內容協作者：我提供想法、素材與判斷，AI 幫我整理文章檔案、檢查格式、更新網站並發布。這個工作方式對我而言，比維護一個傳統後台更接近真正的需求。

## 這個網站是怎麼完成的？

以下是這次實際採用的流程，也是一份可以重做的簡化教學。

### 1. 從 Astro 官方 Blog template 開始

先使用官方部落格範本建立基本專案。它已經包含文章頁、內容處理與基本元件，比從空白資料夾開始更穩定。

### 2. 把文章與作品分成兩個內容集合

文章放在 `content/posts/`，作品放在 `content/works/`。每一篇內容都使用 Markdown，並在檔案開頭提供標題、摘要、日期與標籤。

Astro 的 content collections 會依照預先定義的格式驗證資料。如果漏掉必要欄位，網站會在發布前提出錯誤，而不是讓不完整內容直接上線。[Astro Content Collections 文件](https://docs.astro.build/en/guides/content-collections/)

### 3. 建立首頁、Blog、Works 與 About

首頁顯示最新文章；Blog 是完整文章列表；Works 使用三欄響應式卡片；About 則預留為獨立頁面。這些區塊彼此分開，未來更新其中一頁，不需要回頭重做整個結構。

### 4. 建立品牌視覺

網站採用深紫黑底、淡紫主色與米白文字，字體使用 Noto Sans TC。刻意保持簡潔，讓文章和作品成為畫面主體。

### 5. 補上 SEO 與 GEO 基礎

每篇文章的 frontmatter 會自動產生標題、摘要、作者、日期、標籤、canonical URL、Open Graph 與 Article 結構化資料。網站也會自動產生 sitemap 與 RSS。

GEO（Generative Engine Optimization）不是塞入更多關鍵字，而是讓內容更容易被搜尋系統與生成式 AI 正確理解。因此文章會提供：

- 開頭直接回答核心問題。
- 清楚的標題階層。
- 可比較的表格。
- 有順序的實作步驟。
- 明確的結論與常見問題。
- 可追溯的官方來源。

### 6. 用 GitHub 保存並自動發布

網站原始碼與內容保存在 GitHub。每次更新 `main` 分支後，GitHub Actions 會自動檢查並建置，再由 GitHub Pages 發布。GitHub Pages 也支援日後改用自己的網域。[GitHub Pages 自訂網域文件](https://docs.github.com/en/pages/configuring-a-custom-domain-for-your-github-pages-site/about-custom-domains-and-github-pages)

### 7. 讓 AI 成為內容工作流程的一部分

之後新增文章時，我不一定要自己處理每個技術步驟。我可以先說明主題、觀點、素材和不希望出現的內容，再讓 AI：

1. 整理文章架構與初稿。
2. 補齊 SEO 標題、摘要、標籤與結構。
3. 將內容放進正確集合。
4. 建置檢查。
5. 發布並確認公開網址。

不過，AI 不會替我決定「我是誰」。最終的觀點、經驗與是否發布，仍然要由我負責。

## 這個做法的缺點是什麼？

這套方法並不是沒有代價。

- 沒有像 WordPress 一樣的視覺化文章後台。
- 每次修改都需要重新建置，雖然通常只需要幾十秒。
- 複雜留言、會員或購物功能，之後需要額外服務。
- 如果完全不使用 AI，也不熟悉 GitHub，手動工作流程會比平台編輯器多幾步。

但這些缺點對我目前的內容規模還可以接受。更重要的是，文章是普通 Markdown 檔案，網站程式碼也保存在自己的 GitHub repository。即使未來換工具，內容仍然容易取出與搬移。

## 這個網站本身，就是第一件作品

以前我可能會把「建立作品集」理解成：先完成幾個作品，最後再找一個地方放上去。

這一次順序反過來了。從釐清需求、比較平台、決定架構、建立品牌視覺、整理內容，到把網站真正發布，每一個選擇本身就是作品的一部分。

所以第一篇文章記錄的是「我為什麼做這個網站」，第一個作品則是「我如何把這個網站做出來」。兩者不是重複，而是同一個過程的兩種觀看方式：文章呈現思考，作品呈現結果。

## 常見問題

### 使用 Astro 寫部落格需要會寫程式嗎？

自己從零建立與維護時，需要基本的前端與 Git 知識；如果已經有穩定模板並透過 AI 協作，日常新增文章主要是在提供內容、圖片與修改意見。

### 為什麼不只在方格子或 Medium 發文？

它們適合接觸既有讀者與降低發表門檻，但我還需要作品集、個人品牌與更完整的頁面控制。未來仍可將平台當成分發管道，而不是唯一存放地。

### GitHub Pages 是永久免費的嗎？

目前公開 repository 可以使用 GitHub Pages，但服務政策未來仍可能調整。自訂網域需要另外購買，網站內容則可以隨時搬到其他靜態網站託管服務。

### 以後可以加入真正的後台嗎？

可以。Astro 能串接 CMS 或遠端內容來源；等發文頻率、協作者數量或預覽需求增加後，再加入後台會比一開始就承擔完整系統更合理。

## 結語：先建立一個可以持續變好的地方

我選擇 Astro 與 GitHub Pages，不是因為它在所有情況下都勝過方格子、Medium、WordPress 或 Blogger，而是因為它最符合我現在的排序：**內容與品牌掌控權優先、作品與文章整合、維護負擔保持精簡，並把 AI 納入日常工作流程。**

這個網站目前仍然很簡單，但它已經是一個真正上線、可以持續累積的起點。接下來要做的，不是反覆更換平台，而是把值得留下的文章與作品，一件一件放進來。
